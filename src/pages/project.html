TutorMatch

with Samuel Dovbenyuk

1. Names: Your team, the client, and the app (or suite of apps)
Team: Nevathan, Abishan, Adrian, Liyu
Client: Sam Dovbenyuk (student)
Name of Application: Tutor Match
An application that matches tutors with tutees, simplifying the process of finding a suitable tutor

2. Purpose
The "Tutor Match" app is designed to simplify finding and connecting with tutors through a swipe-based interface similar to Tinder. Tutees can swipe right or left on tutors based on criteria such as GPA, major, hourly rate, and past reviews. If both parties swipe right, they are matched and can communicate or schedule sessions. This approach directly addresses the needs identified in the initial client report from Sam Dovbenyuk, who wanted a platform that streamlined the tutor search process, making it faster and more reliable. Sam also noted a gap in the market where existing platforms charge fees for access to tutors. "Tutor Match" solves this problem by offering a free, user-friendly solution that allows students to quickly find and choose tutors based on key academic and financial factors, without the additional costs.

3. Data Description
Users: To store basic information for all users (tutors and clients). It will include user_id (Primary Key, INT), name (VARCHAR(100)), email (VARCHAR(100)), password (VARCHAR(255)), role (ENUM('tutor', 'client')), and created_at (TIMESTAMP).
Tutors: To store tutor-specific details. It will include tutor_id (Primary Key, INT), user_id (INT), GPA (FLOAT), hourly_rate (FLOAT), bio (TEXT), and subjects (VARCHAR(255)).
Clients: To store client-specific details. It will include client_id (Primary Key, INT), user_id (INT), preferred_subjects (VARCHAR(255)), and budget (FLOAT).
Matches: To track matches between tutors and clients. It will include match_id (Primary Key, INT), tutor_id (INT), client_id (INT), matched_at (TIMESTAMP), and status (VARCHAR(20)).
Swipes: To track swipes made by clients on tutors. It will include the swipe_id (Primary Key, INT), client_id (INT), tutor_id (INT), direction (VARCHAR(10)), and swiped-at (TIMESTAMP).
Reviews: To store reviews and ratings for tutors. It will include review_id (Primary Key, INT), session_id (INT), rating (INT), comment (TEXT), and reviewed_at (TIMESTAMP)

4. Functions performed by the app
Create User Profile (INSERT) - Users provide their name, age, courses, GPA, and other relevant information to create their account and select whether they are creating a tutor or tutee account
Update user profile (UPDATE) - Users can update relevant profile information
Swipe on Tutors (SELECT) - Tutees can swipe to view tutors view information about them and decide if they want to connect with them.
Delete Profile (Delete) - Users can delete their profile

5. Role of PHP and JavaScript
The client side of the app will be built using ReactJS, which allows us to create components for parts of our app. This will include tutor profile cards, swipe functionality, and various pages such as profile setup, browsing tutors, and messaging. React’s state management will help track user interactions, like swipes and matches, in real time without needing to reload the page. Additionally, React’s routing will ensure smooth navigation between pages, creating a seamless experience. JavaScript will handle tasks such as rendering tutor profiles dynamically, responding to swipes or button clicks, and providing instant UI feedback when a match is made. It will also manage form validation for user registration, login, and profile setup to ensure all information is properly entered before being sent to the server.
On the back end, PHP will be responsible for managing the database, storing user information, and running the core logic that connects tutors and tutees. When a user swipes on a profile or updates their preferences, PHP will process this data and determine potential matches based on criteria like subject expertise, location, and price. PHP will also manage and handle updates to profiles and match statuses. It will work with the front end by receiving requests from JavaScript, processing the data, and sending back responses in formats like JSON for the front end to interpret and render.

8. Client Feedback
During our presentation, we introduced TutorMatch as a platform for tutors and tutees to connect. Still, our client Sam emphasized that while the basic concept was solid, it needed to offer something more innovative to stand out. Sam remarked, "I like the idea of a meeting platform, but you need to offer something more innovative than what everyone else is doing," and he stressed that "just connecting tutors and tutees isn’t enough; there has to be a distinctive feature that sets us apart." Taking his feedback into account, we decided to add swipe functionality, where tutees can swipe left or right based on their tutor preferences, including factors such as pay rate and education. Additionally, we are planning to integrate an AI-driven matching algorithm to personalize connections and introduce a dynamic rating and pay system that rewards tutors based on positive feedback from tutees. We are also committed to developing an engaging, user-friendly interface that works seamlessly on both mobile and desktop devices. While TutorMatch is still in development, these planned enhancements are designed to transform it from a simple meeting platform into a dynamic and innovative ecosystem.
